## 🎯 核心设计原则

### 1. 实时性优先
- WebSocket全双工通信
- 游戏状态实时同步
- 低延迟行动响应

### 2. 状态一致性
- 单一数据源（后端游戏引擎）
- 乐观更新与冲突解决
- 断线重连状态恢复

### 3. 安全性
- JWT认证与会话管理
- WebSocket连接验证
- 输入验证与消毒

### 4. 可扩展性
- 模块化游戏引擎
- 插件化WebSocket处理器
- 配置驱动参数调整

## 🎮 游戏引擎架构

### 状态机设计
```
等待中 (waiting)
    ↓ (房主开始游戏)
翻牌前 (preflop)
    ↓ (所有玩家行动完成)
翻牌 (flop)
    ↓ (所有玩家行动完成)
转牌 (turn)
    ↓ (所有玩家行动完成)
河牌 (river)
    ↓ (所有玩家行动完成)
摊牌 (showdown)
    ↓ (分配筹码)
等待中 (waiting)
```

### 核心组件

#### 1. Table (牌桌)
- 玩家座位管理
- 牌堆和公共牌管理
- 下注池计算
- 庄家位置轮转

#### 2. Player (玩家)
- 筹码管理
- 手牌状态
- 行动状态
- 连接状态

#### 3. GameEngine (游戏引擎)
- 游戏流程控制
- 行动处理
- 胜负判定
- 筹码分配

#### 4. PokerEvaluator (牌型评估)
- 使用 treys 库
- 手牌强度评估
- 胜负比较
- 牌型分类

#### PlayerHand (玩家手牌)
- 玩家会话关联
- 手牌ID
- 两张底牌
- 弃牌状态
- 当前下注

## 🛡️ 安全设计

### 认证与授权
1. **JWT认证**
   - 访问令牌：短期有效，用于API调用
   - 会话令牌：长期有效，用于断线重连

2. **WebSocket连接验证**
   - 连接时验证JWT
   - 会话令牌绑定WebSocket连接
   - 心跳保持连接活跃

3. **权限控制**
   - 房主权限：开始游戏、重置游戏
   - 玩家权限：加入/离开牌桌、行动
   - 旁观者权限：观看、聊天

### 数据安全
1. **输入验证**
   - 所有API输入使用Pydantic验证
   - WebSocket消息结构验证
   - SQL注入防护

2. **状态隔离**
   - 玩家只能看到自己的底牌
   - 旁观者可以看到所有底牌（上帝视角）
   - 信息广播按角色过滤

3. **防作弊机制**
   - 服务器端游戏状态验证
   - 行动顺序强制
   - 超时自动处理

## 🔄 断线重连机制

### 会话保持
```
玩家连接 → 创建UserSession → 存储会话状态
    │
    ├─ 正常游戏 → 更新会话状态
    │
    ├─ 断线 → 标记为断开，保留状态
    │
    └─ 重连 → 验证会话令牌 → 恢复状态
```

### 状态恢复流程
1. 玩家使用会话令牌重新连接
2. 服务器验证令牌有效性
3. 恢复玩家的牌桌位置和筹码
4. 如果游戏进行中，恢复手牌状态
5. 如果轮到行动，开始行动计时

### 异常处理
- **行动时断线**：自动弃牌
- **旁观时断线**：保持旁观状态
- **房主断线**：房主权限转移

## 📊 经济系统设计

### 筹码管理
```
初始筹码: 20,000
    │
    ├─ 常规买入: 20,000 (新玩家/中途加入)
    │
    ├─ 重新买入: 中位数 × 50% (破产玩家，限1次)
    │
    └─ 破产处理: 转为旁观者 (第二次破产)
```

### 边池计算
```
玩家A: 全下 1000
玩家B: 下注 1500
玩家C: 下注 2000

主池: 1000 × 3 = 3000
边池: (1500-1000) × 2 = 1000 (玩家B和C竞争)
边池: (2000-1500) × 1 = 500 (只有玩家C)
```
